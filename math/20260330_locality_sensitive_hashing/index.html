Locality Sensitive Hashing

An interactive explainer — from Indyk & Motwani (1998) to LLM inference (2025)

The core problem: Given a dataset of $n$ high-dimensional vectors, find the nearest neighbor to a query point. Exact methods like k-d trees run in $O(2^d \log n)$ — exponential in dimension $d$. At $d = 100$, this is worse than brute force.

The key insight of LSH: Instead of searching exactly, accept a small approximation. If the true nearest neighbor is at distance $r$, find any point within distance $c \cdot r$ (for small constant $c > 1$). This breaks the curse of dimensionality — query time becomes $O(n^\rho \cdot d)$ with $\rho < 1$.

LSH works by constructing hash functions that are similarity-sensitive: nearby points collide in the same bucket with high probability, while distant points rarely collide. Candidate neighbors come only from matching buckets — a tiny fraction of the dataset.

⚠️ LSH ≠ one algorithm. It is a framework. The specific hash family depends on your distance metric: MinHash for Jaccard, SimHash for cosine, p-stable projections for Euclidean. Each has different collision probabilities and performance tradeoffs.

📐 The Math

The $(r_1, r_2, p_1, p_2)$-Sensitive Family

A family of hash functions $\mathcal{H} = \{h : \mathcal{M} \to S\}$ over metric space $(\mathcal{M}, d)$ is $(r_1, r_2, p_1, p_2)$-sensitive if for all $a, b \in \mathcal{M}$ and $h$ drawn uniformly from $\mathcal{H}$:

$$\text{If } d(a,b) \leq r_1 \implies \Pr[h(a) = h(b)] \geq p_1$$ $$\text{If } d(a,b) \geq r_2 \implies \Pr[h(a) = h(b)] \leq p_2$$

with the strict requirement $p_1 > p_2$ and approximation factor $c = r_2 / r_1 > 1$.

Intuition: A hash function is "locality sensitive" if it respects distances. Points close in the original metric end up in the same bucket more often than points that are far apart. The larger the gap $p_1 - p_2$, the better the hash family distinguishes near from far.

The canonical picture: imagine drawing a random bucket boundary through the space. Nearby points are more likely to fall on the same side; distant points are more likely to be separated. LSH formalizes this intuition for arbitrary metric spaces.

The $\rho$ Parameter — Query Complexity

The central performance metric of any LSH scheme is:

$$\rho = \frac{\log(1/p_1)}{\log(1/p_2)} \in (0, 1)$$

An ANN data structure built from this family achieves:

  • Query time: $O(n^\rho \cdot d)$
  • Space: $O(n^{1+\rho})$

Since $\rho < 1$, query time is sublinear in $n$ — the core win over brute force. For example, with $c = 2$ and a good hash family achieving $\rho = 0.5$, querying 1 million points costs roughly $O(\sqrt{n})$ — about 1,000 comparisons instead of 1,000,000.

Historical progress on $\rho$:
Datar et al. (2004): $\rho = 1/c$ for $L_2$
Andoni & Indyk (2006): $\rho = 1/c^2$ — provably near-optimal

Query complexity $n^\rho$ as a fraction of $n$. Smaller $\rho$ means faster queries. The 2006 result ($\rho = 1/c^2$) substantially improves over 2004 ($\rho = 1/c$) especially for small approximation factors.

Probability Amplification: AND–OR Construction

A single hash function rarely separates $p_1$ and $p_2$ enough for practical use. Two composition steps fix this:

AND composition (k functions concatenated)

Define $g(x) = (h_1(x), \ldots, h_k(x))$ — match requires all $k$ to agree.

$$p_1' = p_1^k \qquad p_2' = p_2^k$$

Since $p_1 > p_2$, the gap $p_1^k / p_2^k = (p_1/p_2)^k$ grows exponentially with $k$. But both probabilities shrink, so we need the OR step to recover sensitivity.

OR composition (L independent tables)

Build $L$ independent hash tables. Report a candidate if any table has a match.

$$P_{\text{near}} = 1 - (1 - p_1^k)^L \to 1$$ $$P_{\text{far}} = 1 - (1 - p_2^k)^L \ll 1$$

Choose $k = \log_{1/p_2}(n)$ and $L = n^\rho$. The resulting structure has the promised query time $O(n^\rho \cdot d)$.

The ANN Query Algorithm

Preprocessing (once)

  1. Draw $L$ independent composite hash functions $g_1, \ldots, g_L$, each $g_i(x) = (h_{i,1}(x), \ldots, h_{i,k}(x))$
  2. Build $L$ hash tables $T_1, \ldots, T_L$
  3. Store each point $p$ in bucket $g_i(p)$ of table $T_i$ for all $i$

Query for point $q$

  1. For each table $T_i$, compute $g_i(q)$ and retrieve bucket contents
  2. Union all retrieved candidates across $L$ tables
  3. Compute exact distances to each candidate
  4. Return closest candidate found
The true nearest neighbor $p^*$ collides with $q$ in at least one table with probability $\geq 1 - (1-p_1^k)^L$, which approaches 1 by construction. False positives appear rarely (probability $p_2^k$ per table), so the candidate set stays small.

🔑 Hash Families

The choice of hash family determines which similarity metric LSH approximates. Each family provides a concrete collision probability formula.

MinHash — Jaccard Similarity

Inventor: Andrei Broder (AltaVista, 1997) for near-duplicate web page detection.

Metric: Jaccard similarity over sets: $J(A,B) = \dfrac{|A \cap B|}{|A \cup B|}$

Hash function: For a random permutation $\pi$ of the universe $\mathcal{U}$:

$$h_\pi(S) = \min_{x \in S}\, \pi(x)$$
Core theorem: $$\Pr_\pi[h_\pi(A) = h_\pi(B)] = J(A, B)$$

The proof is elegant: $h_\pi(A) = h_\pi(B)$ if and only if the element with the smallest $\pi$-value lies in $A \cap B$, not just $A \setminus B$ or $B \setminus A$. Among all elements in $A \cup B$, the fraction that lie in $A \cap B$ is exactly $J(A,B)$.

MinHash is a perfect unbiased estimator of Jaccard similarity — no approximation error in the single-hash case, only variance. Using $k$ independent hash functions reduces variance to $O(1/\sqrt{k})$.

Implementation note: avoiding actual permutations

Permuting a universe of size $|\mathcal{U}|$ is expensive. In practice, use a universal hash function family:

h_{a,b}(x) = ((a·x + b) mod p) mod n

where $a, b$ are random integers, $p$ is a large prime. This simulates a random permutation with high probability. The resulting MinHash values are then the minimum over all elements $x \in S$.

Characteristic Matrix Picture

Represent documents as columns in a 0/1 matrix where rows = shingles (elements). Jaccard = count of (1,1) rows divided by count of (1,1) + (1,0) + (0,1) rows.

Still dominant in 2025 for sparse Jaccard similarity: document deduplication, recommendation systems with sparse interaction matrices, genomics sketching (Mash, sourmash). No other method matches MinHash on sparse set similarity.

SimHash / Hyperplane LSH — Cosine Similarity

Inventor: Moses Charikar (2002, STOC).

Metric: Cosine distance: $d_{\cos}(u, v) = \theta(u,v) / \pi$ where $\theta$ is the angle.

Hash function: Choose a random unit vector $r$ from a spherically symmetric distribution:

$$h_r(u) = \text{sign}(r \cdot u) \in \{+1, -1\}$$
Core theorem: $$\Pr_r[h_r(u) = h_r(v)] = 1 - \frac{\theta(u,v)}{\pi}$$

Geometric intuition: A random hyperplane with normal $r$ separates $u$ and $v$ if and only if it passes between them. The fraction of all orientations that split them equals $\theta/\pi$ — the angle they span as a fraction of the semicircle.

$k$-bit SimHash signature: Draw $k$ independent random vectors $r_1, \ldots, r_k$. Compute $(\text{sign}(r_1 \cdot u), \ldots, \text{sign}(r_k \cdot u)) \in \{0,1\}^k$. The expected Hamming distance between signatures of $u$ and $v$ is proportional to $\theta(u,v)/\pi$.

Google's web crawler uses SimHash to detect near-duplicate web pages. With a 64-bit signature, pairs differing in $\leq 3$ bits are flagged as near-duplicates — an $O(1)$ check at crawl scale.

Why sign($r \cdot u$)?

Imagine the unit sphere in $\mathbb{R}^d$. Project $u$ and $v$ onto the sphere (normalize). A random hyperplane through the origin divides the sphere into two hemispheres. The probability that $u$ and $v$ end up in different hemispheres equals the fraction of great circles passing between them — which equals $\theta(u,v)/\pi$.

This is exact for any dimension $d$ — no approximation from the hash function itself, only statistical variance from using finite $k$.

p-Stable Distribution LSH — Euclidean / $L_p$ Distance

Inventors: Datar, Immorlica, Indyk, Mirrokni (2004, SCG).

Metric: Euclidean ($L_2$) distance, or more generally $L_p$ norms.

Hash function: Draw $a \sim \mathcal{N}(0, I_d)$ (i.i.d. Gaussian entries) and $b \sim \text{Uniform}[0, W]$:

$$h_{a,b}(v) = \left\lfloor \frac{a \cdot v + b}{W} \right\rfloor$$

Intuition: Project $v$ onto a random line (direction $a$), then discretize into bins of width $W$. Two nearby points project to nearby values on the line and are likely to land in the same bin. The parameter $W$ controls the bucket width.

Collision probability: For two points at Euclidean distance $\|u - v\| = c$: $$\Pr[h(u) = h(v)] = \int_0^W \frac{1}{c} f_2\!\left(\frac{t}{c}\right)\!\left(1 - \frac{t}{W}\right) dt$$ where $f_2$ is the density of the $|$standard normal$|$ (2-stable distribution). This is a decreasing function of $c/W$ — larger distances → lower collision probability.

The achieved $\rho$ for $L_2$ with this scheme is $\rho \approx 1/c$ (where $c$ is the approximation factor). Andoni & Indyk (2006) improved this to the near-optimal $\rho = 1/c^2$ using a more sophisticated hash family based on the 24-dimensional Leech lattice.

Why Gaussian projections?

A $p$-stable distribution $\mathcal{D}$ has the property: if $X_1, \ldots, X_d \sim \mathcal{D}$ i.i.d. and $c_1, \ldots, c_d$ are real scalars, then $\sum_i c_i X_i \sim \|c\|_p \cdot X$ for $X \sim \mathcal{D}$.

For $p=2$ (Euclidean), the 2-stable distribution is the Gaussian. This means $a \cdot (u - v) \sim \|u-v\|_2 \cdot \mathcal{N}(0,1)$ when $a \sim \mathcal{N}(0, I)$. The projected difference is thus proportional to the $L_2$ distance — exactly what we want.

For $p=1$ (Manhattan), use Cauchy-distributed $a_i$ (1-stable distribution).

The Johnson–Lindenstrauss Connection

Lemma (Johnson & Lindenstrauss, 1984): For any $0 < \varepsilon < 1$ and any $N$ points in $\mathbb{R}^d$, there exists a linear map $f: \mathbb{R}^d \to \mathbb{R}^k$ with $k = O\!\left(\dfrac{\log N}{\varepsilon^2}\right)$ such that for all pairs $u, v$:

$$(1-\varepsilon)\|u-v\|^2 \leq \|f(u)-f(v)\|^2 \leq (1+\varepsilon)\|u-v\|^2$$

The map $f$ can be a random orthogonal projection (or random Gaussian matrix scaled by $1/\sqrt{k}$). This guarantees $O(\log N)$ dimensions suffice to approximately preserve all pairwise distances — independent of the original dimension $d$.

Connection to LSH: JL embedding provides a preprocessing step. After projecting to $k = O(\log N / \varepsilon^2)$ dimensions, standard p-stable LSH is applied. The Fast JL Transform reduces hash computation from $O(d)$ to $O(d \log d)$. Effectively: JL reduces the dimensionality problem, then LSH handles the search problem.

Why does random projection preserve distances?

A random Gaussian matrix $A \in \mathbb{R}^{k \times d}$ (entries i.i.d. $\mathcal{N}(0, 1/k)$) has the property that each row is a random unit-ish vector. The projection $Av$ replaces each coordinate of $v$ with a random linear combination of all coordinates.

For any fixed vector $u - v$, the random variable $\|A(u-v)\|^2$ is a sum of $k$ i.i.d. random variables each with mean $\|u-v\|^2/k$. By concentration of measure (sub-Gaussian tail bounds), the sum concentrates sharply around its mean $\|u-v\|^2$ with deviation $O(\varepsilon \|u-v\|^2)$ when $k = \Omega(\log N / \varepsilon^2)$.

The S-Curve: Banding Amplification

A single hash function gives a linear relationship between similarity and collision probability — not sharp enough to distinguish near from far pairs. The banding trick transforms this into an S-shaped curve with a tunable threshold.

Formula: Divide the $n$-element signature into $b$ bands of $r$ rows each. Two items become candidates if they match exactly in at least one band: $$P(\text{candidate} \mid \text{similarity} = s) = 1 - (1 - s^r)^b$$ Threshold (inflection point): $t \approx (1/b)^{1/r}$. The curve steepens as $b \cdot r$ (total hash functions) increases.

🤖 ML Applications

Approximate Nearest Neighbor Search

The primary ML use of LSH: finding similar embeddings (BERT, CLIP, text-embedding-ada-002, etc.) among millions of vectors without $O(N \cdot d)$ brute-force comparison.

FAISS IndexLSH (Meta AI): flat binary index where vectors are hashed to binary codes compared via Hamming distance. However, FAISS's own team notes that IVF and HNSW indexes consistently outperform LSH on dense embeddings in recall-speed tradeoffs.

⚠️ LSH has largely been displaced for dense embedding search by HNSW and IVF-PQ, which achieve better recall-latency curves in practice. LSH's advantage is formal theoretical guarantees — HNSW is a graph heuristic with no approximation proof.

MagicPIG: LSH for LLM Inference (ICLR 2025)

Uses SimHash to sparsify the KV-cache attention operation during LLM decoding. Key tokens are sampled only when they share a hash bucket with the query in $\geq 2$ of $L$ tables. Result: 5× decoding throughput on Llama-3.1-8B with 96k context on an RTX 4090, achieving 54 ms latency — demonstrating LSH's continued relevance for attention approximation in long-context LLMs.

Recommendation Systems

Recommendation is one of LSH's most established industrial applications, dating to its earliest use at AltaVista.

Core pattern: Collaborative Filtering with LSH

  1. Represent users/items as sparse vectors over the interaction space (e.g., a user's 100-item purchase history over a 10M-item catalog)
  2. MinHash compresses each vector to a $k$-element signature preserving Jaccard similarity
  3. LSH banding identifies candidate similar users/items without $O(N^2)$ comparisons
  4. Collaborative filtering only operates on the candidate pool
Why MinHash specifically? User-item interaction matrices are extremely sparse (a user interacts with $\ll 0.01\%$ of items). Jaccard similarity naturally handles this: $J(\text{user}_A, \text{user}_B) = \frac{|\text{shared items}|}{|\text{union of items}|}$. MinHash is perfect for this. Dense embedding methods (HNSW) are less suitable for sparse binary vectors.

Complexity improvement

With $N$ users, naive user-user similarity requires $O(N^2 \cdot d)$ comparisons — infeasible at $N = 10^8$. MinHash + LSH reduces this to near-$O(N)$ candidate generation, with exact similarity computed only for the candidate pool.

Real-world pipeline

  1. Shingle users: represent each user as the set of items they interacted with
  2. MinHash signatures: compute $k=128$ MinHash values per user (~1KB per user for $10^8$ users = 100GB total, vs. raw interaction matrix ≫ 1TB)
  3. LSH index: build band-based hash tables for fast lookup
  4. At query time: hash the target user's signature, retrieve candidate similar users from matching buckets
  5. Re-rank: compute exact Jaccard or cosine similarity for the $\sim$50–200 candidates
  6. Recommend: aggregate items from the top-$k$ most similar users
MinHash-based LSH for collaborative filtering is still widely deployed at web scale in 2025. It handles the sparse, high-dimensional nature of user-item interaction data better than dense embedding methods.

Near-duplicate content filtering

Recommendation systems also use MinHash to de-duplicate item catalogs — preventing the same article from appearing multiple times with slight rephrasing. Google's web crawler has used SimHash for web-scale dedup since ~2007.

Reformer: LSH Attention in Transformers

Paper: "Reformer: The Efficient Transformer" — Kitaev, Kaiser, Levskaya. ICLR 2020.

Problem: Standard Transformer self-attention computes $QK^T$ for all pairs, giving $O(L^2 \cdot d)$ cost. For $L = 64{,}000$ tokens (book-length), this is intractable.

Key insight: In dot-product attention, query $q_i$ only receives large weight from keys $k_j$ with high cosine similarity to $q_i$ — the attention distribution is sparse. Attending only within LSH buckets approximates full attention at $O(L \log L)$ cost.

The tie trick: Standard attention has independent $W_Q \neq W_K$. Reformer ties $W_Q = W_K$ so queries and keys come from the same projection. This is necessary: LSH can only find similar vectors if queries and keys are in the same space.

Angular LSH procedure

  1. L2-normalize each query/key vector (project onto unit sphere)
  2. Sample random rotation matrix $R \in \mathbb{R}^{d \times b/2}$
  3. Compute $xR$, concatenate with $-xR$ → shape $[d, b]$
  4. $\text{bucket}(x) = \text{argmax}(\text{concat}(xR,\, -xR))$
  5. Repeat with different $R$ for multiple hash tables

Attention computation cost vs sequence length. LSH attention achieves the $O(L \log L)$ scaling that makes book-length sequences tractable.

Additional Reformer innovations

  • Reversible residual layers: eliminate need to store $N$ layer activations during backprop, reducing memory from $O(N \cdot L \cdot d)$ to $O(L \cdot d)$
  • Chunked FFN layers: compute feed-forward in chunks to avoid large intermediate tensors

Together, these let the Reformer process sequences up to 1 million tokens on a single 16GB accelerator.

Reformer limitations: Tied Q/K is architecturally restrictive. Stochastic bucket assignment requires multiple hash rounds. For sequences $< 2048$ tokens, standard attention is faster. The Reformer has largely been superseded by FlashAttention (IO-aware exact attention) and sliding-window attention in production — but it introduced the powerful concept of using LSH to sparsify attention.

When to Use LSH vs Alternatives (2025)

Use LSH when:
  • Sparse vectors + Jaccard similarity — MinHash is unbeaten for document dedup, sparse recommendation CF, genomics sketching
  • Formal approximation guarantees required — HNSW is a heuristic; LSH has proven $(r_1, r_2, p_1, p_2)$-sensitivity
  • Very high-dimensional sparse data ($d > 10{,}000$, sparsity $> 99\%$) — IVF/HNSW degrade here
  • Memory-constrained environments — no graph structure overhead vs HNSW
  • LLM attention sparsification — MagicPIG (ICLR 2025) shows 5× gains on long-context inference
  • Near-duplicate detection at crawl scale — SimHash remains Google's method of choice
Use HNSW when:
  • Dense embeddings (BERT, CLIP, OpenAI embeddings)
  • High recall required (>95%)
  • Real-time insertions/deletions needed
  • Moderate scale (tens to hundreds of millions of vectors)
  • Sufficient RAM available
Use IVF-PQ when:
  • Billion-scale datasets
  • Disk-resident or memory-constrained at scale
  • Filtered similarity search
  • Batch query workloads
